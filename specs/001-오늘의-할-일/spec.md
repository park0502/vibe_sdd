# 기능 명세서: 오늘의 할 일 웹 앱

**기능 브랜치**: `001-오늘의-할-일`

**작성일**: 2026-07-15

**상태**: Draft

**입력**: 사용자 설명: "오늘의 할 일" 웹 앱을 만들려고 해. 사용자 관점의 요구사항은 다음과 같아.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 할 일을 추가하고 관리한다 (우선순위: P1)
사용자는 새 할 일을 입력하고 목록에 추가할 수 있다. 입력이 비어 있거나 공백만 있으면 추가되지 않으며, 안내 메시지를 받아 확인할 수 있다. 이후 할 일을 완료 처리하거나 삭제할 수 있다.

**왜 이 우선순위인가**: 이 흐름이 핵심 사용자 가치의 시작점이며, 할 일 앱의 기본 사용 경험을 구성한다.

**독립 테스트**: 새 할 일을 입력해 목록에 추가하고, 완료/삭제 동작을 수행해도 앱이 정상적으로 반응하는지 하나의 시나리오로 검증할 수 있다.

**수용 시나리오**:
1. **Given** 사용자가 할 일 입력창에 내용을 입력한 상태일 때, **When** 사용자가 추가 동작을 수행하면, **Then** 목록에 새 할 일이 추가된다.
2. **Given** 사용자가 입력창에 빈 값 또는 공백만 입력한 상태일 때, **When** 사용자가 추가 동작을 수행하면, **Then** 새 할 일은 추가되지 않고 안내 메시지가 표시된다.
3. **Given** 목록에 등록된 할 일이 있을 때, **When** 사용자가 해당 항목을 완료 처리하면, **Then** 항목이 완료 상태로 표시되고 상태에 따라 시각적으로 구분된다.

---

### 사용자 스토리 2 - 할 일을 완료/미완료 상태로 전환하고 삭제한다 (우선순위: P1)
사용자는 할 일 항목을 완료 또는 미완료로 토글할 수 있고, 삭제 전 확인 절차를 거쳐 실수로 삭제하는 상황을 방지할 수 있다.

**왜 이 우선순위인가**: 완료 상태 전환과 삭제는 사용자가 할 일을 관리하는 핵심 동작이므로 핵심 기능으로 본다.

**독립 테스트**: 할 일을 완료 상태로 전환한 뒤 다시 미완료로 되돌리고, 삭제 확인 절차를 통해 삭제가 실제로 수행되는지 확인할 수 있다.

**수용 시나리오**:
1. **Given** 완료되지 않은 할 일이 있을 때, **When** 사용자가 완료 토글을 수행하면, **Then** 항목은 완료 상태로 바뀐다.
2. **Given** 완료된 할 일이 있을 때, **When** 사용자가 다시 토글을 수행하면, **Then** 항목은 미완료 상태로 되돌아간다.
3. **Given** 삭제하려는 할 일이 있을 때, **When** 사용자가 삭제를 시도하면, **Then** 삭제 전 확인 절차가 표시되고 사용자가 확인해야만 삭제가 진행된다.

---

### 사용자 스토리 3 - 목록 상태를 확인하고 필터링한다 (우선순위: P2)
사용자는 남은 할 일 개수를 확인하고, 전체/진행중/완료 필터를 사용해 목록을 원하는 범위로 좁혀 볼 수 있다.

**왜 이 우선순위인가**: 진행 상황을 한눈에 파악하는 기능은 할 일 관리의 효율성을 높여주므로 중요하다.

**독립 테스트**: 상태 요약과 필터 조건이 바뀔 때마다 표시되는 목록이 올바르게 달라지는지 검증할 수 있다.

**수용 시나리오**:
1. **Given** 완료되지 않은 할 일이 여러 개 있고 완료된 할 일이 하나 있을 때, **When** 사용자가 화면을 확인하면, **Then** 남은 할 일 개수가 올바르게 표시된다.
2. **Given** 필터가 전체 상태일 때, **When** 사용자가 진행중 필터를 선택하면, **Then** 완료되지 않은 항목만 표시된다.
3. **Given** 필터가 전체 상태일 때, **When** 사용자가 완료 필터를 선택하면, **Then** 완료된 항목만 표시된다.

---

### 엣지 케이스

- 입력값이 비어 있거나 공백만 있을 때는 항목이 추가되지 않아야 한다.
- 완료 상태와 미완료 상태가 섞여 있을 때 남은 항목 수와 필터 결과가 올바르게 반영되어야 한다.
- 할 일이 하나도 없을 때는 목록 대신 빈 상태 안내를 보여주어야 한다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 사용자가 새 할 일을 입력해 목록에 추가할 수 있게 해야 한다.
- **FR-002**: 시스템은 입력값이 비어 있거나 공백만 있을 때 새 할 일을 추가하지 않고 사용자에게 안내 메시지를 표시해야 한다.
- **FR-003**: 시스템은 사용자가 할 일을 완료 상태로 전환할 수 있게 해야 한다.
- **FR-004**: 시스템은 완료된 할 일을 다시 미완료 상태로 되돌릴 수 있게 해야 한다.
- **FR-005**: 시스템은 사용자가 할 일을 삭제할 수 있게 해야 하며, 삭제 전에 확인 절차를 거쳐야 한다.
- **FR-006**: 시스템은 화면 상단에 남은 할 일 개수를 항상 표시해야 하며, 추가/완료/삭제 동작 후 즉시 갱신되어야 한다.
- **FR-007**: 시스템은 전체, 진행중, 완료 3가지 필터를 제공해야 한다.
- **FR-008**: 시스템은 브라우저 새로고침 또는 서버 재시작 후에도 할 일 목록이 유지되도록 저장해야 한다.
- **FR-009**: 시스템은 모바일 화면(360px 폭)에서도 사용할 수 있도록 반응형 레이아웃을 제공해야 한다.
- **FR-010**: 시스템은 할 일이 하나도 없을 때 친절한 빈 상태 안내 문구를 표시해야 한다.
- **FR-011**: 완료된 할 일은 시각적으로 구분되어야 하며, 취소선과 흐린 색을 사용해 표시할 수 있다.

### 핵심 엔티티 *(데이터가 포함되므로 포함)*

- **할 일 항목(Todo Item)**: 사용자가 관리하는 할 일 단위로, 텍스트 내용, 완료 여부, 식별자를 포함한다.
- **할 일 목록(Todo List)**: 여러 할 일 항목을 담는 집합으로, 필터 상태와 현재 남은 개수를 함께 관리한다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 사용자는 새 할 일을 1분 이내에 입력하고 목록에 추가할 수 있어야 한다.
- **SC-002**: 사용자가 할 일 추가, 완료 토글, 삭제, 필터 변경을 수행한 후에도 목록 상태와 남은 개수는 즉시 반영되어야 한다.
- **SC-003**: 사용자는 할 일이 하나도 없을 때 빈 상태 안내를 보고 다음 행동을 이해할 수 있어야 한다.
- **SC-004**: 브라우저 새로고침 또는 서버 재시작 후에도 사용자가 기존 할 일 목록을 계속 확인할 수 있어야 한다.
- **SC-005**: 360px 폭 화면에서도 주요 기능과 텍스트가 모두 접근 가능해야 한다.

## 가정

- 사용자는 기본적인 웹 브라우저 사용 경험을 가진다.
- 초기 버전에서는 사용자 계정이나 동기화 기능은 포함되지 않는다.
- 데이터 저장 방식은 브라우저의 로컬 저장소(localStorage)를 사용한다. 이를 통해 새로고침이나 브라우저 재시작 후에도 할 일 목록을 유지한다.
- 기술 스택은 계획 단계에서 결정한다.
