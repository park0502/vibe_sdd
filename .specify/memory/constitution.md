# vibe_sdd Constitution

## 핵심 원칙

### 1. 스펙 우선
모든 기능은 구현 전에 스펙 문서로 정의한다. 스펙에 명시되지 않은 기능은 구현하지 않는다. 변경이 필요하면 먼저 스펙을 갱신한 뒤 구현을 진행한다.

판정 기준:
- 구현 전에 스펙 문서가 존재한다.
- 코드 변경 사항이 스펙에 포함된 범위 안에 있다.
- 스펙에 없는 기능이 구현되면 위반으로 간주한다.

### 2. 테스트 필수
핵심 기능인 할 일 추가, 완료 토글, 삭제, 남은 개수 계산, 필터 기능은 pytest 자동화 테스트로 검증한다. 구현 완료의 기준은 해당 테스트가 통과하는 것이다.

판정 기준:
- 핵심 기능에 대한 pytest 테스트가 작성되어 있다.
- 테스트가 통과한 결과를 확인할 수 있다.
- 회귀가 발생했을 때 테스트가 실패한다.

### 3. 설정 분리
데이터베이스 경로와 같은 환경 설정 값은 .env 파일로 관리하고 저장소에 커밋하지 않는다. 필요한 환경변수는 .env.example 파일에 문서화한다.

판정 기준:
- 민감하거나 환경별로 달라지는 값이 .env에만 저장된다.
- .env.example에 필요한 변수명과 예시가 기록되어 있다.
- 저장소에 .env 파일이 커밋되면 위반으로 간주한다.

### 4. 추적 가능한 커밋
모든 커밋 메시지는 작업 식별자를 포함한 형태로 작성한다. 예시는 "T03: 할 일 생성 API 구현"과 같다.

판정 기준:
- 커밋 메시지 앞에 태스크 ID가 포함되어 있다.
- 태스크 ID가 의미 있는 작업 단위와 일치한다.
- 커밋 메시지가 작업 내용을 설명한다.

### 5. 단순성
요구사항에 없는 기능이나 라이브러리는 임의로 추가하지 않는다. 해결해야 할 문제를 가장 단순한 방식으로 구현한다.

판정 기준:
- 새 기능이나 의존성 추가가 스펙 또는 현업 필요에 근거한다.
- 불필요한 추상화나 과도한 설계가 없다.
- 추가된 라이브러리가 실제 필요성을 가진다.

## 추가 제약
이 프로젝트에서 생성되는 모든 문서(constitution, spec, plan, tasks, checklist, 분석 리포트 등)는 반드시 한국어로 작성한다.

판정 기준:
- 생성된 문서의 본문이 한국어로 작성되어 있다.
- 외부 의존성 없이 이해 가능한 한국어 표현을 사용한다.
- 문서 제목과 섹션도 한국어 기준으로 관리한다.

## 개발 절차
모든 구현 작업은 스펙 확인, 테스트 작성, 구현, 검증 순서로 진행한다. 변경 사항이 생기면 관련 문서와 테스트를 함께 갱신한다.

판정 기준:
- 새 기능 작업 전에 스펙과 테스트 범위가 정리되어 있다.
- 구현 후 테스트와 문서 검증이 수행된다.
- 코드 변경과 문서 변경이 분리되지 않는다.

## Governance
이 Constitution은 이 프로젝트의 우선 규칙이며, 모든 개발 활동은 이 원칙과 일치해야 한다. 변경 사항은 문서와 테스트를 함께 갱신하고, 리뷰 과정에서 준수 여부를 확인한다.

개정 절차:
- 원칙 변경 또는 새 규칙 추가는 관련 문서와 함께 제안한다.
- 변경 제안은 리뷰를 거쳐 합의된 뒤 반영한다.
- 주요 변경은 버전을 올리고 수정 이력을 기록한다.

준수 검토:
- PR 또는 변경 사항 리뷰 시 위 원칙을 기준으로 확인한다.
- 테스트가 없는 변경은 기본적으로 반려 대상으로 본다.
- 문서 작성 기준을 위반한 변경은 보완이 필요하다.
